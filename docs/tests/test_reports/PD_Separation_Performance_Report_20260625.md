## 3. 延迟分解分析

```
短提示-短输出 (500ms 稳态延迟分解):

Prefill阶段: ~200ms (40%)
  ├── Token处理 + Embedding: ~50ms
  ├── 28层Attention + MLP: ~140ms  
  └── KV Cache计算: ~10ms

KV传输阶段: ~1.5ms (<1%) ☆ 极优
  ├── 序列化: ~0.3ms
  ├── P2P传输: ~0.9ms
  └── 反序列化: ~0.3ms

Decode阶段: ~298ms (60%)
  ├── 20 tokens × ~14.9ms/token
  └── Decode吞吐: 67 tok/s (单请求时)
```

## 4. 性能瓶颈分析

### 4.1 瓶颈识别

#### 瓶颈1: Decode阶段吞吐量 (主瓶颈 ⚠️)

| 场景 | Decode吞吐 | 评价 |
|------|-----------|------|
| 单请求 | 40-67 tok/s | 中等 |
| 10并发 | 7.4 tok/s/req | **严重不足** |

**原因**:
- TP=2 每卡仅承载一半decoder层
- Qwen2-VL每token需计算28层，计算量大
- 并发时串行处理，无批处理加速

**优化**: 增加Decode节点TP或加实例

#### 瓶颈2: 并发串行化 (⚠️)

| 单请求延迟 | 10并发延迟 | 放大倍数 |
|------------|-----------|----------|
| 500ms | 6776ms | **13.5x**  |

**原因**: Proxy轮询 → 单Prefill + 单Decode → 请求串行排队

**优化**: 多实例 + 缓存感知调度

#### 瓶颈3: 冷启动延迟

首次请求11749ms vs 稳态500ms = **23.5x慢**

**原因**: KV传输连接建立 + 首次计算图编译

**优化**: 启动时预热

### 4.2 非瓶颈项 ✓

| 组件 | 性能 | 结论 |
|------|------|------|
| **KV传输** | 1.5ms | 极优, 可忽略 |
| **模型加载** | 5.3s | 启动一次性 |
| **内存使用** | 24.6GB/32GB | 充足 |
| **Prefix缓存** | 可达45%+ | 可用 |

## 5. 优化建议

### 5.1 立即执行 (高优先级)

#### 优化1: Decode节点扩展

**方案**: TP=4 或 多Decode实例
**预期**: Decode吞吐 2x, 并发延迟 -50%

```bash
# 方案A: 将Decode改为TP=4
export ASCEND_RT_VISIBLE_DEVICES=4,5,6,7
--tensor-parallel-size 4

# 方案B: 增加Decode实例数
--num-decoders 2
```

#### 优化2: 请求预热

```bash
# 启动后立即发送预热请求
curl -s http://127.0.0.1:8000/v1/completions \
  -d '{"model":"...", "prompt":"warmup", "max_tokens":1}'
```

**预期**: 消除冷启动延迟

### 5.2 短期 (1-2周)

#### 优化3: 缓存感知调度

```python
# 当前: 轮询选择Prefiller
# 优化: Prefix-hash感知选择

def select_prefiller(request):
    prefix_hash = hash(request.prompt[:128])
    for p in prefillers:
        if p.has_cached(prefix_hash):
            return p  # 缓存命中 → 跳过Prefill
    return least_loaded()
```

**预期**: Prefix Hit Rate 45% → 70%+

#### 优化4: 增大批处理窗口

```
--max-num-batched-tokens 4096 → 8192
```

**预期**: 并发吞吐 +30-50%

### 5.3 长期 (1-2月)

- **动态扩缩容**: 基于负载自动增删实例
- **多节点PD分离**: 跨节点部署支持大规模
- **KV量化传输**: FP8量化 → 传输带宽减半
